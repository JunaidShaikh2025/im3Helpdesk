1. Product Overview

iM3 Helpdesk is a multi-tenant helpdesk SaaS where each customer is an Organization. Each organization subscribes to one of three plan tiers and pays per agent seat per month (or annually with a discount). Approval-based payments are currently used; gateway-based real payments are planned in Phase 2.

Key principles

  • One organization = one active subscription at any time.
  • Every new organization gets a 14-day free Growth trial on signup.
  • Feature access is gated by the plan tier — both in the UI and on the API.
  • SuperAdmin manages pricing and approves payments; CompanyAdmin manages their org's subscription.

5. Subscription Lifecycle

  1. Signup → new organization created → 14-day Growth trial auto-issued (SubscriptionSeeder).
  2. Trial banner on dashboard for CompanyAdmin only, with dynamic color: blue (>7d) → amber (4–7d) → red (≤3d).
  3. Upgrade → CompanyAdmin opens Explore Plans, picks a higher tier, fills payment dialog, submits.
  4. Approval → SuperAdmin reviews in Payment Approvals queue (notification badge shown), then approves or rejects with notes.
  5. Activation → on approve: previous subscription is cancelled, a new OrganizationSubscription is created with status Active and the new period.
  6. Downgrade → blocked in UI (Upgrade button hides) and on API (400 Plan downgrade is not allowed. Please contact support.). Reason: refund/proration complexity in MVP.

6. Current Payment Flow — Manual Approval (Phase 1, live)

This is what is already implemented and running today.

StepActorWhat happens
1CompanyAdmin Clicks Upgrade to X on Explore Plans → dialog asks for seats, billing cycle, billing name/email/address, card last 4 + brand (informational).
2Backend POST /api/subscription/payments creates a PaymentRecord with status Pending.
3SuperAdmin Sees pending count badge in sidebar → opens Payment Approvals → reviews details → Approves or Rejects.
4Backend On approve: cancels current sub, creates new active sub for the chosen tier & cycle, marks payment Approved.
5CompanyAdmin Refreshes — features unlock instantly. Plans & Billing shows new plan + payment in history.
Why manual? No money actually changes hands inside the app yet — useful for staged rollout, soft-launch with hand-picked customers, or off-platform settlements (bank transfer / UPI screenshot share).

7. Real Payment — Gateway Integration Plan (Phase 2)

7.1 Why a gateway

Bank account numbers and card details must never be stored in the app (PCI-DSS & RBI regulations). Instead we plug in a payment gateway that collects the money on our behalf and settles it to the SuperAdmin's registered bank account.

7.2 Recommended gateway

ProviderBest forCost (approx)Settlement
RazorpayIndia-first SaaS2% per txnT+3 days to current account
StripeGlobal / USD pricing2.9% + ₹3T+7 days
CashfreeIndia alt., subscriptions1.75% per txnT+1 to T+2

Recommendation: Razorpay for v1 — simplest SDK, INR-native, easy KYC.

7.3 SuperAdmin one-time setup (outside the app)

  1. Create a Razorpay business account at razorpay.com.
  2. Complete KYC — PAN, GST, address proof, current bank account.
  3. From Razorpay Dashboard → Settings → API Keys, generate Key ID & Key Secret.
  4. Webhook secret & URL: https://YOUR-DOMAIN/api/payments/webhook/razorpay
The bank account lives only in Razorpay, never in our DB. Settlements (collected money minus fees) land there automatically.

7.4 In-app SuperAdmin screen — Payment Settings (new)

  • Sidebar: Super Admin → Payment Settings
  • Fields: Provider, Key ID, Key Secret (write-only, masked when re-shown), Webhook Secret, Test/Live toggle.
  • "Test connection" button hits a sandbox order create.
  • Stored in new table PaymentGatewaySettings; secret column encrypted via ASP.NET Data Protection.

7.5 New customer flow (replaces manual approval for cards/UPI)

  1. CompanyAdmin clicks Upgrade → frontend calls POST /api/subscription/checkout.
  2. Backend creates a Razorpay Order and returns { orderId, keyId, amount }.
  3. Frontend opens Razorpay Checkout popup with that order.
  4. Customer pays (card / UPI / netbanking / wallet).
  5. Razorpay calls POST /api/payments/webhook/razorpay → backend verifies signature → marks payment Approved + activates subscription.
  6. Frontend listens to razorpay.success and redirects to Plans & Billing.
Effect: No manual SuperAdmin approval needed — activation happens in seconds. Manual queue stays available for offline (NEFT/UPI) cases.

10. Security & Compliance

  • Never store full card numbers, CVVs, or bank account numbers in our DB. Only the masked last 4 + brand are persisted for receipts.
  • Gateway secrets (Key Secret, Webhook Secret) stored encrypted using ASP.NET Data Protection keyring.
  • Authorization: payment submission requires CompanyAdmin role; approval queue requires SuperAdmin role.
  • Audit: every payment row carries SubmittedByUserId, ReviewedByUserId, timestamps, and notes.
  • Webhook hardening (Phase 2): signature verification, idempotency key (gateway payment id), replay protection.
  • HTTPS-only in production; webhook endpoint will allow only the gateway's IP range.
  • PCI-DSS scope stays minimal because card data is collected by the gateway iframe, not our app (SAQ-A).
